# 简历分析多Agent助手项目日志
## 2026/5/11
**事件**：

因为上次实现知乎心理学Agent写作助手的固定流程编排，决定挑战自己，在本项目里引入高一个复杂层级的架构：固定流程编排+单点路由。

结合本人的实际需求，因为要求职，所以想写一个自动判断自己简历是否符合JD要求的Agent助手，而且这个需求也适用于HR筛选海量的求职简历，具有一定的商业价值，所以就决定开发简历分析助手。

而流程编排可以参考之前开发的知乎心理学Agent项目，再稍微加入LLM路由，实现单点路由。

**结果**：

敲定项目架构，具体流程编排。

## 2026/5/12
**事件**：

大概敲定需求方向和流程编排后，找ChatGPT按照构思生成几个具体的实现方案，然后选出一个架构不复杂的轻量Agent项目文件目录框架。

**问题**：

发现ChatGPT有本能的生成企业级别的文件目录倾向，如果不严格限制Agent项目文件目录设计的各种可能性，ChatGPT给的架构都是复杂的，而且会把功能文件拆的特别细，很影响本人拿到文件目录后，整体把握这个设计方案的思路。

**解决**：

先询问ChatGPT在Agent项目方面都有哪些技术方向，架构设计，然后把这些方面作为约束加入项目生成提示词，反复的调试，LLM输出的质量明显提升。

**心得**：

LLM难以理解抽象的约束，要严格限制LLM必须写出具体限制哪些方面。

## 2026/5/13
**事件**：

昨天让LLM生成具体的设计方案后，要求LLM遵循MVP模型，一步一步生成最小可用项目，在前一个项目上逐步迭代，于是先辅助开发这个项目的前端代码，并且生成打桩测试的后端，先调通前端和后端。

**问题**：

LLM给出的前端代码有一些复杂，并且运行后也没有一个明确的验收标准，没有第一步的MVP的退出标准。

**解决**：

让LLM在生成代码时就配套生成测试代码，并且明确要求LLM给出清晰的测试退出标准，方便拿到后一键测试。

**事件**：

完成固定流程编排，先不引入路由节点和研究分支，然后跑通这个测试。

## 2026/5/14
**事件**：

加入路由节点，并且通过测试。

**事件**：

加入研究节点。

**问题**：

测试出现报错：SMOKE_TEST_RESULT: FAILURE - [direct] route mismatch: expected direct, got revise。

原本以为是后端graph.py文件里，可能没有把执行的信息正确返回给smoker_test.py，但实际上是LLM路由有问题，因为路由的决策权完全在LLM手上，一开始我默认我给的简历信息和JD一定会让LLM路由输出走revise的路径，但实际上LLM有时会走，有时不会走，导致我调试这个BUG非常的莫名其妙，一会测试可以过，一会又不可以过。

还好一开始要求LLM在生成代码的时候必须生成执行日志，根据日志定位了问题的原因。

**解决**：

优化了路由LLM的提示词，当检查到用户提示词出现强烈要求联网检索时，则必定会输出revise路径，经过十次测试，效果稳定。

## 2026/5/15
**事件**：

进一步优化前端页面，丰富用户的使用体验。

**问题**：

前端莫名其妙变成空白的深蓝色界面。

**解决**：

翻了翻浏览器控制台，发现是让LLM生成的前端优化代码忘记导入import React from 'react'。

**突发事件**：

DeepSeek的API莫名其妙访问超时，导致无法正常完成LLM路由。

**解决**：

增加了_fallback_rout函数，当兜底机制，自动补充json文件里缺乏的值。

# 碎片知识
1. LLM会在一开始的最小迭代版本上，加入很多打桩测试，来实现最小闭环，但是在随后的更新，如果你没有提醒LLM更新原先的打桩测试，则LLM有可能不会主动更新。
2. 
3. 在项目开发后，如果自己引入新的依赖包，一定要告诉LLM，否则它仍然会默认使用原先的依赖。
4. 
5. 借用LLM开发项目，必须让LLM设置日志，方便显示各个阶段的报错，更快debug。
